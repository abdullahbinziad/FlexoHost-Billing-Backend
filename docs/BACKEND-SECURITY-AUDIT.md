# Backend Security Audit Report

## Summary

This document covers security findings, malware/attack scope, and mitigations for the FlexoHost Billing backend.

---

## HIGH SEVERITY (Fixed)

### 1. **ReDoS – Unescaped Regex in Search Queries**
**Locations:** `activity-log.service.ts`, `promotion.service.ts`, `product.service.ts`

**Issue:** User-controlled search strings were passed directly to `$regex` or `new RegExp()`. Malicious input like `(a+)+$` or `(a|a?)+` can cause catastrophic backtracking (ReDoS), leading to CPU exhaustion and DoS.

**Fix:** Use `escapeRegex()` from `utils/escapeRegex.ts` before any user input in regex. Applied to:
- Activity log search
- Promotion search (code, name)
- Product name in create/update duplicate check

---

### 2. **Path Traversal in deleteFile**
**Location:** `user.util.ts` – `deleteFile(filePath)`

**Issue:** `filePath` from the database was used with `path.join(process.cwd(), filePath)` without validation. A compromised or malicious value like `../../../etc/passwd` could delete files outside the app directory.

**Fix:** Normalize path, strip leading `../`, resolve, and ensure the result is under `process.cwd()` before deleting. Only delete if the target is a file.

---

### 3. **File Upload – MIME Type Spoofing**
**Location:** `middlewares/upload.ts`

**Issue:** The previous filter used a regex on both extension and mimetype. Clients can spoof `Content-Type`, so a malicious file could be accepted by matching a fake mimetype.

**Fix:** Strict allowlist mapping extension → allowed MIME types. Both extension and mimetype must match. Reject if they don’t align.

---

## MEDIUM SEVERITY

### 4. **Malware in Uploaded Files**
**Status:** Partially mitigated

**Issue:** No virus/malware scanning. Malicious content in allowed types (e.g. macro-enabled DOC, malicious PDF) can be uploaded and distributed via ticket attachments.

**Recommendations:**
- Add ClamAV or a cloud malware scanner for uploads
- Consider content inspection (magic bytes) for critical flows
- Serve uploads with `Content-Disposition: attachment` and safe MIME types to reduce auto-execution

---

### 5. **WHMCS Migration – SQL Execution**
**Location:** `whmcs-migration.service.ts`

**Issue:** Uploaded SQL is executed directly. Malicious SQL in the file could affect the MySQL database. Access is restricted to admin/superadmin.

**Mitigations:**
- Route protected by `restrictTo('admin', 'superadmin')`
- File size limit (200MB)
- Only `.sql` and `.zip` allowed

**Recommendation:** Run migration in an isolated environment or use a dedicated migration tool with stricter controls.

---

### 6. **import-sql.ts – Command Injection**
**Location:** `scripts/whmcs-migration/import-sql.ts`

**Issue:** Uses `execSync` with paths from `process.argv[2]` or `WHMCS_SQL_PATH`. If an attacker can control these, command injection is possible.

**Mitigation:** Script is for local/admin use. Paths are validated with `path.resolve` and `fs.existsSync`. Still, avoid passing user input into shell commands.

---

## LOW SEVERITY / INFORMATIONAL

### 7. **Static File Serving**
**Location:** `app.ts` – `express.static(uploadPath)`

**Status:** Safe. `express.static` prevents path traversal. Uploaded filenames are generated by multer (no user-controlled path).

---

### 8. **Notification linkPath**
**Status:** Safe. `linkPath` is built server-side from IDs (e.g. `/tickets/${id}`). No user input. Frontend also validates before navigation.

---

### 9. **NoSQL Injection**
**Status:** Mitigated. `express-mongo-sanitize` strips `$` and `.` from user input.

---

### 10. **Rate Limiting**
**Status:** In place. Global limit and stricter auth limit (20 req/15 min) for login/register/forgot-password.

---

### 11. **CORS**
**Status:** Configured. Origin from `config.cors.origin`.

---

### 12. **Helmet**
**Status:** Enabled for security headers.

## Malware / Attack Scope

| Vector              | Risk   | Mitigation                                      |
|---------------------|--------|-------------------------------------------------|
| ReDoS               | High   | `escapeRegex()` on all search inputs            |
| Path traversal      | High   | Path validation in `deleteFile`                  |
| MIME spoofing       | Medium | Extension ↔ MIME allowlist in upload           |
| Malware in uploads  | Medium | No scanner; consider ClamAV                     |
| SQL injection       | Low    | Migration is admin-only; SQL from trusted dump  |
| Command injection   | Low    | import-sql is admin-only; path validation       |
| NoSQL injection     | Low    | mongo-sanitize                                  |
| XSS                 | Low    | API returns JSON; frontend sanitizes HTML       |
| Mass assignment     | Low    | Whitelisting in controllers                     |

---

## Implemented Fixes (Remaining Gaps)

1. **ClamAV malware scanning** – Optional virus scan for uploads. Set `ENABLE_CLAMAV_SCAN=true` and install ClamAV. Integrated in ticket attachments, avatar, and generic upload routes.
2. **HttpOnly cookie-only auth** – Set `COOKIE_ONLY_AUTH=true` to omit tokens from API response. Frontend uses `credentials: 'include'` and no longer stores tokens when in cookie-only mode.
3. **Secrets validation** – Startup check warns if JWT secrets appear weak in production.
4. **npm audit** – Run `npm audit fix` regularly. Some dev dependencies (minimatch, puppeteer) may require `--force` for full resolution.

## Recommendations

1. Use `Content-Disposition: attachment` for non-image uploads.
2. Run `npm audit` regularly and fix vulnerabilities.
3. Ensure `.env` is never committed and production secrets are strong.
